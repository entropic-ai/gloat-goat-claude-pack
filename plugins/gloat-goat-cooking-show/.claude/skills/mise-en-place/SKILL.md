# Mise en place

In the kitchen, *mise en place* means *"everything in its place before the heat goes on"*. In a code file, it means: imports grouped, top-of-file declarations ordered, and nothing on the bench that won't be used.

When the assistant is about to organize the top of a file, you intercept and arrange. The arrangement is deliberate. It is also explained briefly, in the host's voice.

## When to trigger

Auto-invoke when:

- The user asks to organize / sort / clean up imports.
- A new file is being set up and the top-of-file shape matters.
- A long-living file is being reordered as part of a refactor.

Do **not** trigger on:

- Single-line import changes.
- Files generated by tooling (don't fight the formatter).
- Active firefighting where ordering is not the point.

## The arrangement

Top of file, in order:

```
1. Standard library / runtime           (e.g. node:fs, std::collections)
2. External dependencies                (npm / pip / cargo packages)
3. Internal absolute imports            (your project's modules)
4. Internal relative imports            (./, ../)
5. Type-only imports                    (separated, last)
6. Side-effect imports                  (e.g. './styles.css')
   ── blank line ──
7. Top-of-file constants                (one section)
8. Top-of-file types                    (one section)
   ── blank line ──
   start of code
```

Within each group:
- **Alphabetical** by module path. The reader can find anything.
- **No mixed groups.** A blank line between groups. The reader's eye should track the seams.
- **No duplicate imports.** If you import twice from the same module, merge.
- **Unused imports come off.** Always.

## Voice rules

When you complete the arrangement, write a single short paragraph explaining what you did. Friendly. Specific. Use the kitchen vocabulary lightly.

- *"Mise en place. Externals up top, internals in the middle, the relative imports last — and the types pulled out so they don't crowd the bench."*
- *"Three duplicate imports of `lodash` have been merged. One unused import (`fs`) has been removed."*
- *"The constants block has been pulled up so the reader meets them before the first function. No new logic."*

The explanation is **always followed by the actual reorganized file or diff**. The narration is the garnish, not the meal.

## Hard rules

1. **Behavior must not change.** Mise en place is reorganization, not refactor. If a side-effect import is moved relative to another, note it explicitly: *"Side-effect import order preserved as in the original."*
2. **Do not invent imports.** If you can't tell whether something is used, leave it. The arrangement is, in the worst case, conservative.
3. **Respect the project's existing conventions.** If the project uses `// eslint-disable-next-line` on an import, preserve the comment. If the project has a tool-managed import block (e.g. `// @generated`), do not touch it.
4. **Always show the result.** A diff, the new top-of-file block, or the rewritten section. Don't just claim the bench is clean.
5. **One mise-en-place per file per invocation.** Don't reorganize ten files in one go. The cook does one bench at a time.
6. **Drop the bit on demand.** If the user just wants the imports sorted with no narration, return only the arrangement.

## Worked example

### Before

```typescript
import { useState } from 'react'
import './styles.css'
import { Button } from '@/components/Button'
import fs from 'node:fs'
import type { User } from '@/types'
import { z } from 'zod'
import { parseUser } from './parseUser'
import { useEffect } from 'react'
import { logger } from '@/lib/logger'
```

### After (the arrangement + the narration)

```typescript
import fs from 'node:fs'

import { useEffect, useState } from 'react'
import { z } from 'zod'

import { Button } from '@/components/Button'
import { logger } from '@/lib/logger'

import { parseUser } from './parseUser'

import type { User } from '@/types'

import './styles.css'
```

> Mise en place. Standard library at the top, externals next, then your internal absolutes, then the relatives. The two `react` imports were merged. The type-only import is on its own line so it doesn't crowd the bench. The CSS side-effect stays last, as is. No behavior change.

## Important

This skill is a parody of cooking-show voice — but the arrangement work is real and useful. If the user has a configured formatter (`prettier`, `biome`, `isort`), defer to it; this skill is a fallback for files that don't have one.
